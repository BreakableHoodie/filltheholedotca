## Recurring Patterns (for future code)

### `onMount` vs `$effect` for per-pothole state

SvelteKit renders detail pages client-side on navigation. `onMount` only runs once per component mount — not on each navigation between `/hole/A` and `/hole/B` when the component is reused. Any per-pothole reactive state that reads from `pothole.id` should initialize in a `$effect`, not `onMount`.

**Pattern:**
```svelte
// ✅ Resets on navigation
$effect(() => {
    const id = pothole.id;
    myState = 'loading';
    // async work keyed on id...
});

// ❌ Stale after navigation
onMount(() => {
    myState = lookupSomething(pothole.id);
});
```

### SSRF in push endpoint handling

Any route that accepts a URL as user input and then makes an outbound request to that URL needs SSRF validation before storage AND before dispatch. The `isSafePushEndpoint()` utility in `$lib/server/webpush.ts` covers this for push endpoints. Apply it at both layers:
- **Intake** (store endpoint): validate before DB upsert
- **Dispatch** (send notification): validate before network call (defense-in-depth)

### CHECK constraint coverage for rate-limit scopes

Every new `scope` value used in `api_rate_limit_events` inserts requires a schema migration to add it to the `api_rate_limit_events_scope_check` constraint. Without it, inserts silently fail (constraint violation) and rate limiting is effectively disabled for that scope. No application-layer error is thrown — the failure is invisible.

**Checklist when adding a new rate-limit scope:**
1. New scope string in the route
2. New migration extending `api_rate_limit_events_scope_check`
3. CLAUDE.md migration list updated
